# Deliverables System Documentation

> See also: [EXECUTABLE-PIPELINES.md](./EXECUTABLE-PIPELINES.md) for pipeline architecture (verified)

## Overview

The deliverables system enables users to create AI-powered code changes through a web interface. It connects repository selection, task definition, file attachments, and real-time pipeline execution.

## System Architecture

### Data Flow
```
User Input → API Route → Pipeline Execution → SSE Stream → UI Updates
```

### Component Stack

#### Frontend (Unified entry at `uapi/app/executions/`)
- **page.tsx** (executions): Unified entry that drives behavior via `type` param and reuses the Deliverables page without remount
- `uapi/app/executions/components/ExecutionPageClient.tsx`: Main client logic with access control and execution flow (rendered by /executions)
- **useTaskState**: SSE streaming and state management hook
- **VCSSourceSelectors**: Repository selection with auto-select
- **ProcessLog**: Accordion-based execution log display
- **Execute Button**: Primary action with cancel support

#### API Layer (Unified wrappers under `uapi/app/api/executions/*`)
- **POST `/api/executions?type=pipeline:deliverables`**: Execute pipeline (streamed)
- **GET `/api/executions?type=pipeline:deliverables`**: VCS resource listing (owner/repos/branches/commits/files via action/params)
- **GET `/api/executions/stream?type=pipeline:deliverables&runId=...`**: SSE stream for real‑time updates
- **GET `/api/executions/history?type=pipeline:deliverables`** and **`/history/{runId}`**: Execution history and details
- **DELETE `/api/executions/{runId}?type=pipeline:deliverables`**: Cancel running pipeline
Note: Legacy `/api/deliverables/*` routes remain as implementation; unified `/api/executions/*` delegates to them to avoid behavior changes.

#### Pipeline (`packages/pipelines/deliverable/`)
- SDIVS pattern execution (Setup → Discovery → Implementation → Validation → Shipping)
- ALL agents use PTRR (Plan-Try-Refine-Retry) - no exceptions
- FOUR deliverable types:
  1. **Code Change** - Creates Pull Request (Divide|Conquer|Correct pattern)
  2. **Code Change Review** - Reviews Pull Request
  3. **Design Document** - Creates Issue
  4. **Design Document Review** - Comments on Issue
- Dynamic agent registration based on deliverable type determination
- Complete E/E hierarchy with proper execution classes

## Complete Pipeline Architecture (Production E2E)

### Current Implementation SSOT (EE‑based)
This section reflects the current source implementation using execution‑generics (EE) “sequence and save” primitives and registry‑backed agent executors.

- Setup (PhaseRunner; declarative):
  - Agents (in order):
    1) `setup:deliverable-pipeline-clone-vcs-repository-agent`
    2) `setup:deliverable-setup-plan-agent`
    3) parallel: `setup:deliverable-pipeline-comprehend-task-agent` (+ optional `setup:deliverable-pipeline-initialize-lsp-agent`)
  - Repo normalization (workspacePath/owner/name/branch/provider/connectionId) is performed by the clone agent wrapper (closest to source), not by the phase code.

- Discovery (EE sequential executor):
  - `discovery:gather-context` → `discovery:understand-requirements` → `discovery:research-approach` → `discovery:plan-implementation` → `discovery:assess-complexity`.

- Implementation (EE executor; dynamic per type):
  - `code-change`: Divide → dynamic parallel Conquer (one agent per file) → Correct.
  - `code-change-review` | `design-document` | `design-document-review`: single PTRR agent.
  - Conquer returns patch/new content only; validation occurs in Correct/Validation.

- Validation (EE sequential executor):
  - Parallel:
    - `validation:validate-last-iterations-validation-phase` → store(`validation/last`,'issues',string[])
    - `validation:validate-discovery-phase` → store(`validation/discovery`,'issues',string[])
    - `validation:validate-implementation-phase-{type}` → store(`validation/implementation`,'issues',string[])
  - Then: `validation:deliverable-pipeline-ready-to-ship-agent` (PTRR; can short‑circuit with refund metadata). Reads issues from validation/* stores.

- Shipping (EE sequential executor; standardized):
  - `shipping:deliverable-pipeline-ship-agent` (PTRR) → type‑aware tool selection (use‑computer + VCS create PR/issue/comment).
  - `shipping:final-work-summary` (Quick) → UI header payload.

Tools SSOT:
- Generic: `use-computer` (packages/generic-tools/use-computer) with deliverables wrapper delegating to it.
- Shipping: VCS tools for PR/issue/comment are registered and mapped in deliverables registry; Ship agent selects via `output.useTools`.

Execution State:
- Populated automatically by AG/EE/PG (phase/agent/step/failsafe/generation; tool invocation/results). Setup phase returns input unchanged; downstream phases read from stores; no manual emitters.

### Validation Stores (issues‑only validators)
- `validation/last:issues` → from `validation:validate-last-iterations-validation-phase`.
- `validation/discovery:issues` → from `validation:validate-discovery-phase`.
- `validation/implementation:issues` → from `validation:validate-implementation-phase-{type}`.

### Short-Circuit Primitive
The executor-level primitive for elegant pipeline termination with automatic refunds:

```typescript
// In execution-generics
export interface ShortCircuitSignal {
  type: 'SHORT_CIRCUIT';
  reason: string;
  refundType: 'full' | 'partial';
  confidence: number;
  metadata?: Record<string, any>;
}

// PipelineExecutor catches these automatically
if (hasShortCircuitSignal(output)) {
  throw new ShortCircuitError(output.signal);
}
```

### Phase Sequence with Short-Circuit Points

#### Setup Phase
1. `clone-vcs-repository` - Clone repo from VCS
2. `initialize-lsp` - Initialize language server
3. `danger-wall` - Safety check (SHORT-CIRCUIT: full refund)
4. `comprehend-task` - Understand user's request
5. `determine-deliverable-type` - Classify into 4 types
6. `analyze-codebase` - Initial codebase analysis
7. `ready-to-iterate` - Go/no-go decision (SHORT-CIRCUIT: full refund)

#### Discovery Phase
1. `comprehend-attachments` - Process Figma/images/docs
2. `select-files-parallel` - Parallel file selection
3. `understand-requirements` - Deep requirement analysis
4. `analyze-parallel` - Parallel file analysis
5. `plan-implementation` - Create implementation plan
6. `assess-complexity` - Complexity assessment

#### Implementation Phase (type-specific)
**Code Change (Divide|Conquer|Correct)**:
- `divide` - Split work into file-level tasks
- `conquer-file-1..N` - Parallel file processing
- `correct` - Fix and harmonize changes

**Code Change Review**:
- `review-pull-request` - Comprehensive PR review

**Design Document**:
- `create-issue` - Generate GitHub issue

**Design Document Review**:
- `add-comment` - Add issue comment

#### Validation Phase
1. `validate-discovery` - Validate discovery outputs
2. `validate-implementation` - Validate implementation
3. `validate-validation` - Meta-validation!
4. `ready-type-specific` - Type-specific readiness
5. `ready-to-ship` - Final decision (SHORT-CIRCUIT: partial refund)

#### Shipping Phase (Standardized)
Two agents only:
1) `shipping:deliverable-pipeline-ship-agent` (PTRR)
2) `shipping:final-work-summary` (Quick)

### Agent Implementation Pattern

Every agent follows the PTRR-7 substep pattern:

```typescript
const agent = factoryAgentWithPTRR<Input, Output>({
  name: 'agent-name',
  outputSchema: OutputSchema,
  plan: { chunkThreshold: 1000 },
  try: { chunkThreshold: 2000 },
  refine: { maxAttempts: 2 },
  retry: { maxAttempts: 1 }
});
```

Each PTRR step automatically runs 7 substeps:
1. PrepareContext (gathers execution state)
2. ChunkThenSum (if content > threshold)
3. StitchUntilComplete (multi-pass coherence)
4. Reason (thinks about task)
5. Judge (evaluates approach)
6. StructuredOutput (generates typed output)
7. Tool execution (if useTools in schema)

### Refund Processing

Located in `@bitcode/credits` package (used by all pipelines):

```typescript
// Automatic refund on short-circuit
withCreditReservation(userId, async (escrow) => {
  // Pipeline execution...
  // If ShortCircuitError thrown, automatic refund
}, { pipelineType: 'deliverable' });
```

#### Divide|Conquer|Correct Pattern (Code Change only)
- **Divide**: Determines ALL files to be edited/created/deleted
- **Conquer**: Parallel execution - one ConquerFile agent per file
- **Correct**: Validates and corrects all file changes
- PrepareContext substep automatically gathers relevant execution state
