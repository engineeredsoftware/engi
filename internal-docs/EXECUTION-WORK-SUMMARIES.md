# Execution Work Updates Architecture

## Overview

This document outlines the design and implementation of the Work Update system for Engi executions. The system provides real-time visibility into AI agent work during pipeline execution, creating an "intelligence escape hatch" where users can see not just what was done, but how and why it was done.

## Customer Needs

### Primary Requirements

1. **Visibility During Execution**: Users need to see what the AI is doing in real-time, not just a final result
2. **Intelligence Transparency**: Show the reasoning process, not just actions taken
3. **Work Attribution**: Clear tracking of which files were changed and which tools were used
4. **Self-Instruction Display**: Surface the AI's self-generated instructions for iteration guidance
5. **Contextual Placement**: Summary area positioned between execution logs and instruction input

### User Experience Goals

- **Non-intrusive**: Summaries complement but don't overwhelm the execution interface
- **Progressive Disclosure**: Collapsed by default with ability to expand for details
- **Real-time Updates**: Stream updates as work progresses
- **Context Preservation**: Transform from "Latest Work" to "Iteration N Work" when iterations complete

## Architecture Design

### Type Hierarchy

```
WorkUpdateBase (abstract)
├── AgentStepWorkUpdate      - After each PTRR step (Plan/Try/Refine/Retry)
└── SDIVSPipelineUpdate      - After a DIV iteration completes

PipelineCompletionSummary (separate concept)
└── DeliverableCompletionSummary - Final deliverable summary (existing FinalWorkSummary)
```

### Key Distinctions

| Aspect | Work Updates | Completion Summary |
|--------|---------------|-------------------|
| **Timing** | During execution | After pipeline completion |
| **Purpose** | Show HOW work is being done | Show WHAT was delivered |
| **Updates** | Real-time streaming | Once at end |
| **Content** | Process and reasoning | Results and metrics |
| **UI Location** | Between logs and instructions | Header of completion page |

## Implementation Components

### 1. Data Types (`packages/execution-generics/src/work-update.ts`)

Core interfaces for work summaries with:
- Narrative prose (generated by LLM)
- Structured file changes with purpose
- Structured tool usage with purpose
- Self-instruction content (from ready-to-instruct agent)
- Timing and performance metrics

### 2. Generation Points

#### Agent Step Postprocessing
- **Location**: `packages/agent-generics/src/steps/factories.ts`
- **When**: After each PTRR step completes
- **How**: LLM call to generate narrative from step output + tool/file tracking

#### Pipeline Iteration Postprocessing
- **Location**: `packages/pipelines/deliverable/src/agents/validation/deliverable-pipeline-ready-to-instruct-agent.ts`
- **When**: After each DIV iteration
- **How**: Ready-to-Instruct agent stores self-instruction text and emits `SDIVSPipelineUpdate`

### 3. Storage Pattern

```typescript
// During execution - work updates
execution.store('work-update', 'latest-agent-step', update);
execution.store('work-update', `iteration-${n}`, update);
execution.store('work-update', 'latest-iteration', update);

// Completion updates remain separate
execution.store('completion-summary', 'final', completionSummary);
```

### 4. Streaming Architecture

Work updates stream automatically via execution state changes:
```typescript
execution.on('store', (namespace, key, value) => {
  if (namespace === 'work-update') {
    stream.write({ type: 'work-update', scope: key, update: value });
  }
});
```

### 5. UI Components

#### WorkUpdatePanel
- **Location**: `/uapi/components/base/engi/execution/WorkUpdatePanel.tsx`
- **Purpose**: Render agent-step and iteration updates with a unified visual design
- **Features**:
  - Prose narrative describing progress
  - File change list with per-file deltas
  - Tool usage breakdown with success/error indicators
  - Iteration variant includes self-instruction text, confidence gauge, and suggestions

#### Integration Points
```tsx
// In ExecutionsPageClient.tsx
<PipelineExecutionLog />
<WorkUpdatePanel variant="latest" update={latestWorkUpdate} />
{iterationUpdates.map(update => (
  <WorkUpdatePanel key={update.id} variant="iteration" update={update} />
))}
<ExecutionsInstructions />

// In ConversationsSidebarLogs.tsx
<PipelineExecutionLog />
<WorkUpdatePanel variant="latest" update={latestWorkUpdate} />

// In ConversationsCard.tsx (embedded)
// Work update integrated into log column
```

## Current Issues & Improvements Needed

### 1. LLM Call Optimization
- Current design makes an LLM call after every step - potentially expensive
- Consider batching or only generating narratives at iteration boundaries
- Could use cheaper models for step summaries, better models for iterations

### 2. Data Flow Complexity
- Multiple aggregation points (step → agent → iteration)
- Need clear ownership of summary generation
- Postprocessing layer responsibilities unclear

### 3. Naming Consistency
- "Work Update" vs "Work Narrative" vs "Execution Summary"
- Should align with existing patterns (e.g., FinalWorkSummary)
- Consider simpler naming: ExecutionProgress?

### 4. Storage Overhead
- Storing full summaries at each level creates redundancy
- Consider storing only deltas or references
- Need retention/cleanup strategy

### 5. UI Polish Needs
- Component design is functional but needs visual refinement
- Expand/collapse animations missing
- Mobile responsiveness not considered
- Accessibility (ARIA labels, keyboard navigation) incomplete

## Recommended Next Steps

### Phase 1: Simplify Core Design
1. Reduce to two update types: `AgentStepWorkUpdate` and `SDIVSPipelineUpdate`
2. Generate narratives only at iteration boundaries (not every step)
3. Use structured data aggregation for step-level tracking

### Phase 2: Optimize LLM Usage
1. Batch multiple steps into single LLM call
2. Use lightweight model for routine summaries
3. Cache and reuse common narrative patterns

### Phase 3: Refine UI/UX
1. Add smooth animations for expand/collapse
2. Implement keyboard navigation
3. Add copy-to-clipboard for summaries
4. Create responsive mobile layout

### Phase 4: Production Hardening
1. Add error boundaries around summary generation
2. Implement fallback for LLM failures
3. Add metrics/monitoring for summary generation
4. Create retention policies for stored summaries

## Open Questions

1. **Streaming granularity**: Should we stream every step or batch updates?
2. **Storage duration**: How long to retain work summaries?
3. **Cost management**: What's acceptable LLM spend per execution?
4. **Feature flags**: Should this be gradually rolled out?
5. **Customization**: Should users control verbosity/detail level?

## Success Metrics

- **Visibility**: % of executions where users expand work summaries
- **Performance**: Summary generation time < 500ms
- **Cost**: LLM costs < $0.01 per execution
- **Reliability**: >99.9% summary generation success rate
- **User Satisfaction**: Positive feedback on execution transparency

## Conclusion

The Work Update system addresses a critical need for execution transparency but requires refinement to be production-ready. The core architecture is sound but implementation complexity and LLM costs need optimization. Focus should be on simplifying the design while preserving the key value: giving users visibility into AI reasoning and work process.
