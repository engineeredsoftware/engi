# ENGI Spec V12 Notes

## What V12 changes relative to V11

V11 made ENGI operationally inevitable.
V12 asks whether the demonstration itself now fully deserves that confidence.

That is a different kind of work.

V12 is not mainly about repairing missing system surfaces.
It is mainly about proving, through the demo, that ENGI’s operating model is the natural shape of generative enterprise needs.

## The main V12 shift

The central shift is:
- from a strong operating picture,
- to a demonstration whose thesis feels complete.

That means the core question becomes:

**Does the depositing-to-needing path feel so native that branch, proof, and settlement follow as obvious consequences?**

## Depositing and needing are now the center

V12 should make these the primary operator concepts:
- depositing,
- needing,
- their fit,
- and the closure path they induce.

The rest of the system matters.
But the demo should not lead with the rest.

## Why V12 should not reopen the whole system

By the end of V11, the core ENGI system shape is already strong:
- repo-bound intake,
- identity/auth spine,
- proof closure,
- exact settlement,
- boundary realism,
- profile distinction by deposit mode and need mode.

So V12 should assume:
- little if any major system spec rescue remains,
- any spec changes discovered now are likely edge-rounding discovered through demonstration refinement.

## Strong V12 naming bias

Prefer names that make the demonstration thesis obvious.

Good V12 examples:
- depositing surface
- needing surface
- depositing-to-needing surface
- repo-to-settlement surface
- identity/auth spine
- boundary reality surface

Avoid names that sound like generic UI composition or product garnish.

## V12 demo-quality questions

A good V12 pass should be able to answer:
1. Does deposit selection feel native, not like a form workaround?
2. Does need measurement feel consequential before deep artifact reading?
3. Is the fit between the deposit and the need obvious?
4. Do different artifact kinds feel genuinely different?
5. Does identity/auth feel like the system spine?
6. Do proof and settlement read as necessary closure rather than appended validation?
7. Does the whole system feel like the canonical operating model for generative enterprise needs?

## The most important V12 risk

The main risk is over-improving the system instead of improving the demonstration.

If a proposed change does not materially improve:
- depositing,
- needing,
- their fit,
- the operator story,
- or the feeling of inevitability,

then it is probably not V12 work.

## Suggested first V12 implementation slice

The highest-value first slice is:
1. foreground depositing and needing in the shell,
2. add an explicit depositing-to-needing fit surface,
3. make profile A/B differences obvious through deposit mode and need mode,
4. keep proof/settlement present but downstream in the explanation,
5. preserve all existing system honesty.
