# @bitcode/doc-code

V26 status: admitted `ingress-or-support` plus `compatibility` corridor for build-time tool prompt attachment.
`@bitcode/doc-code` keeps tool prompt descriptions attached to tool instances so Bitcode agentic runs can consume full tool documentation without each package inventing its own runtime convention.

Runtime prompt injection system - automatically attaches prompt instances to tool instances based on `@doc-code-*` comment patterns.

## What It Does

Doc-code is a build-time transformation system that:
1. Finds `@doc-code-*` comments with `@prompt PROMPTNAME` 
2. Injects `instance.__docCodePrompt = PROMPTNAME` and `instance.__promptParts = PROMPTNAME` after instantiation
3. Makes prompts available at runtime for LLM consumption and prompt-part-aware tooling

## The Pattern

```typescript
import { WEB_SEARCH_DOC_CODE_TOOL_PROMPT } from './prompts/WebSearchPrompt';

/**
 * @doc-code-tool
 * @prompt WEB_SEARCH_DOC_CODE_TOOL_PROMPT
 */
class WebSearchTool extends Tool<typeof search> {
  use = search;
}

export const searchTool = new WebSearchTool();

// After webpack build, automatically becomes:
export const searchTool = new WebSearchTool();

// Attach DocCodeToolPrompt for runtime injection (auto-generated by doc-code-tool-loader)
searchTool.__docCodePrompt = WEB_SEARCH_DOC_CODE_TOOL_PROMPT;
searchTool.__promptParts = WEB_SEARCH_DOC_CODE_TOOL_PROMPT;
```

## Installation

The webpack loader is automatically configured in `uapi/next.config.mjs`:

```javascript
config.module.rules.push({
  test: /\.(ts|tsx)$/,
  include: [
    path.resolve(__dirname, '..', 'packages', 'generic-tools'),
    path.resolve(__dirname, '..', 'packages', 'tools-generics'),
  ],
  use: [{
    loader: path.resolve(__dirname, '..', 'packages', 'doc-code', 'loader.js'),
    options: { exclude: [/\.test\./, /\.spec\./] }
  }]
});
```

## How It Works

1. **Pattern Detection**: Webpack loader scans TypeScript files during build
2. **Regex Matching**: Finds classes with `@doc-code-*` and `@prompt NAME`
3. **Code Injection**: Adds `__docCodePrompt` and `__promptParts` assignment after instantiation
4. **Runtime Access**: Tools can be queried for their documentation

## Current Implementation

- **doc-code-tool**: The current admitted implementation for Tool classes
- Supports any class pattern with instantiation
- Requires the prompt constant to be imported in the same file
- Coexists with the runtime compatibility helper `attachDocCodeToolPrompt` during convergence

## Type Safety

The Tool base class includes the type definition:

```typescript
export abstract class Tool<T extends ToolFunction = ToolFunction> {
  // ... other properties
  
  /**
   * DocCodeToolPrompt attached by build-time transform.
   * @internal
   */
  __docCodePrompt?: any;

  /**
   * Prompt parts compatibility carrier attached alongside __docCodePrompt.
   * @internal
   */
  __promptParts?: any;
}
```

## Usage with formatUsableTools

```typescript
import { formatUsableTools } from '@bitcode/tools-generics';

const tools = [searchTool, getContentsTool];
const documentation = formatUsableTools(tools);
// Returns formatted documentation from attached prompts
```

## Important Notes

- The `@prompt` field must reference an imported constant
- The loader runs automatically during webpack builds
- No manual transformation needed
- Works with Next.js, webpack, and other build tools that support loaders
- This package is an admitted support/compatibility corridor, not an independent product semantics owner
